                            Gary Rowe Mod Flash 6 years ago

                            The article tries to make it clear that this approach is for projects that use a Maven reactor as the mechanism for sharing build and dependency information within a Java project. Often large development teams have developers who prefer one IDE over another (my personal preference is Intellij) and using Maven removes the issue of forcing everyone to use the same IDE - they merely have to agree on the build system which Maven handles extremely well.

                            There is a side issue to this. Maven works very well with a continuous integration (CI) approach that is recommended by the Extreme Programming Agile approach. On GitHub you can use Travis to do this. Enterprises tend to use Jenkins. If you didn't have an IDE-agnostic build process then CI would be rather hard whereas Maven makes it trivial. As an example, try building one of my Dropwizard projects from the command line: https://github.com/gary-row....

                            I'm not an expert on Android, but this article gives a comprehensive guide: http://www.vogella.com/tuto...
